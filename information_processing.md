### 4.1.1. OneDriveを島流しにします
この補助資料の構成を何回考えても、やっぱりOneDriveを消し去らないと話が始まらない気がします。僕が書いた記事を見ながらやれば多分ミスはないはずなので、やっておくことを非常に強く推奨します。もちろん、「脳死で使っててもデータが残る」とはならなくなるので、データが消し飛ばないように気を付けないといけないですが、普通にGoogle Driveなどに大事なファイルは自分で保存したりして対策するのが良いと思います。

<a id="four-one-two"></a>
### 4.1.2. エクスプローラーを使いこなそう ― 覚えることは3つだけ
エクスプローラーというのはパソコンの下の方のバー(タスクバー)の左の方にある水色と黄色の、明らかにファイルを管理しそうなアイコンのやつです。

みんなに安心してほしいので言っておきますが、僕は普段からプログラミングをしますがエクスプローラーも使います。また、エクスプローラーの操作も普通にマウスでカチカチするのでエクスプローラーの操作は僕よりも速い人が沢山いると思います。

また、エクスプローラーの説明と書いたんですけど説明することはほとんどないです。<br>
- 初期設定として **「拡張子の表示」, 「隠しファイルの表示」** をすること。
- 普段使いのための操作として **「困ったらPC→C:→ユーザー→[ユーザー名]」** と **「クイックアクセスへの追加」** と **「↑ボタン(1階層上に戻るボタン)」** <br>

のみです。つまり、普段使いのために覚えるのは3つだけです。

<a id="four-one-two-one"></a>
#### 4.1.2.1. 困ったときは(PC→)C:→ユーザー→[ユーザー名] ― 宇宙から星へ
まず、初心者の人が困ること第1位は「どこに何があるか分からない」だと思います。で、これを1番手っ取り早く解決するには決まった手順を何回も繰り返して覚えることが大事だと思います。

まずエクスプローラーを開きましょう。そうすると、画面の左の方に「ホーム」だの「ギャラリー」だの「デスクトップ」だのがあります。これはナビゲーションウィンドウというらしいですが、僕はサイドバーと呼びます。

- このサイドバーの下にスクロールすると「PC」というのがあります。もしかしたらそのすぐ下に「よく分かんない文字列+(C:)」となってるやつ(今後C:と表記し、Cドライブと読みます)もあるかもしれません。<br>
**もし「PC」しかない場合はそれをクリックして、「よく分かんない文字列+(C:)」がある場合はそっちをクリックします。**<br>
**「PC」にアクセスした人はC:があるはずなのでクリックしてそこにアクセスしてください。**

- 今これで皆さんはC:というところにいるはずです(これは[4.2. ディレクトリの話をしよう](#five-two)で言うWindowsの宇宙の最高司令部ですね)。<br>
そこからは「ユーザー→ [ユーザー名]」と進んで下さい(ユーザー名は自分が一番よく使うやつです、分からないなら順番にアクセスしてどれか確かめましょう)。<br>
下のフォルダが見えていればhome dirという場所(つまり星の最高司令部です、C:と同様に後で説明します)にきちんと進めています。
```text
/C:/Users/<ユーザー名>/ (= ~)
├─ Desktop
├─ Documents
├─ Downloads
├─ Music
├─ Pictures
├─ Videos
├─ AppData
│  ├─ Local
│  ├─ LocalLow
│  └─ Roaming
├─ OneDrive
├─ .local (最初の状態では存在しない)
│  └─ bin  
└─ (そのほかユーザーが作成したフォルダ)
```

- home dirに進めたらまたサイドバーの「PC」や「C:」をクリックして「ユーザー→ [ユーザー名]」と進んで、C:やhome dirに何があるかを覚えるまで繰り返してください。

授業中に暇でかつdir構造を覚えていない方はずっとやってください。これをやった回数がその後のパソコンスキルに繋がってきます。

<a id="four-one-two-two"></a>
#### 4.1.2.2. 拡張子の表示, 隠しファイルの表示 ― 上のバーの方の「表示」で設定
まずは拡張子の表示や隠しファイルの表示をします。
拡張子というのは`.pdf`, `.txt`, `.md`, `.py`, `.ipynb`, `.docx`などで、これは **「このファイルはこういう感じのファイルだよー」みたいなのを伝えるための識別子みたいなもの** です。<br>
隠しファイルの表示はプログラミングに全く興味がない人はやらなくてもいいんですが、やって損はないのでやった方が良いとは思います。また、パソコンでマイクラがしたい方はやった方がいいです。

では実際の手順に移るのですが、とりあえずエクスプローラーを開いてください。そうすると、「クイックアクセス」とか書かれてて「デスクトップ」だの「ダウンロード」だのよく見るフォルダが出てくると思います。

そして、 **そこから上の方に視線を動かすと「表示」とか書かれているのが見えると思います。** そこをクリックすると1番下にさらに「表示」が出てきます。<br>
**「表示」の中の「表示」をさらにクリックする**ことで「ファイル名拡張子」や「隠しファイルの表示」という項目が出てくるのでどちらもクリックしておきましょう。チェックマークがつきます。

<details><summary>隠しファイルとは</summary>
  まず、「隠しファイルを表示」という設定ではありますが、隠しファイルのみならず隠しフォルダも表示されるようになります。<br>
一般的に隠しファイルは一般的には<code>.zshrc</code>, <code>.bashrc</code>, <code>.gitconfig</code>などの<code>.</code>始まりのファイルで、隠しフォルダも同様に<code>.local/</code>や<code>.ssh/</code>などの<code>.</code>始まりのフォルダなどといわれています 。
<strong>が、僕のパソコンdと<code>.</code>はじまりのファイルも隠しファイルを表示しない設定でも普通に表示されてるのでよく分かりません。</strong><br>
この後説明する「パス」でいう<code>/c/Recovery/</code>や　<code>/c/Users/[ユーザー名]/AppData</code>は隠しフォルダとしてWindowsにされており、隠しフォルダを表示にしないと表示されないのですが、それ以外はよく分かんないです。
</details>

#### 4.1.2.3. クイックアクセスにピン留め
では次に、普段使いのための操作である「クイックアクセス」への追加と「1階層上に戻る」ボタンの話をします。

サイドバーを使ってわちゃわちゃします。<br>
例えば「ドキュメント」dirに農工大のファイルを入れるdir(`~/Documents/tuat/`)を作った場合は、
- サイドバーからドキュメント」dirに飛ぶ
- `tuat/`フォルダをクリックして右クリックして「クイックアクセスにピン留めする」をクリックする
- サイドバーを下にスクロールすると追加した`tuat/`dirが追加されているので次回から簡単にアクセスできます。

同じように学期ごとに`1_fyear_fsem`(1年前期, 命名の仕方は[農工大のファイルを分けて整理しよう](#農工大のファイルを分けて整理しよう)に書いています)をなどをクイックアクセスに追加しておくと使いやすいです。

また、デフォルトの状態のサイドバーにピン留めされている「デスクトップ」, 「ダウンロード」, 「ドキュメント」, 「ピクチャ」などなどが全部OneDriveの方のフォルダがピン留めされてる可能性があるんですよね。<br>
(つまり下の記事で言うドッペルゲンガーフォルダです)。

ぜひサイドバーからピン留めされてる「ドキュメント」などにアクセスしてみて、画面の上の方に「OneDirve」って書いてないかを確認してほしいです。
もし「OneDrive」の文字列があるならサイドバーからそのフォルダを右クリックして「クイックアクセスからピン留めを外す」で外して下さい。

そして、`PC`→`C:/Users/[ユーザー名]/`にアクセスしたところにある「ドキュメント」や「デスクトップ」をピン留めするようにしましょう。

#### 4.1.2.4. 階層上に戻るボタン ― 上向きの矢印

すると、上の方に「PC > C: > ユーザー > [ユーザー名]」と書かれてますよね。この部分をアドレスバーとかって言います。<br>
そこから目線を真横に向けると、上向きの矢印がありますよね。<br>
これをクリックしてみましょう。画面はどうなりましたか？

そうですね、「ユーザー」の画面に切り替わったと思います。さらにまた上向きの矢印をクリックしてみましょう。

C:に動いていると思います。<br>
つまり、この上向きの矢印(↑)は1つ上の階層に移ることを意味します。<br>
<details><summary>実は1階層上じゃない？</summary>
  実はこれは厳密には2階層上なんじゃないかという話があるんですよね。
  なんかそのファイルが存在するdir Aが1階層上で、dir Aが入っているdir B  は2階層上なんじゃないかみたいな。例えば、
  Pythonでは<code>Path</code>モジュールで同じdirじゃないdirのファイルを<code>import</code>しようとしたときとかに
  <code>
  str(Path(__file__).parent.parent)
  </code>
  みたいなコードが出てくるんですが、<code>parent</code>、つまり親dirが  2回出てきているのに これは僕たちの言う「1階層上」なんですよね。
  とはいえ、直感的に理解するためにこの資料では1階層上として扱います。
</details>

### 4.1.3. 農工大のファイルを分けて整理しよう
パソコンの良いところは「このファイルはここ」ってその人のやりたいように整理できることです。物理的な制約がないからです。<br>
なので、全てのファイルをてきとーに作っててきとーに管理しないようにしましょう。<br>
ということで、農工大のファイルを整理するとしましょう。プログラミングとかに全く興味がないならこういう感じにするのがいいと思います。
```
C:/Users/[ユーザー名]/Documents/農工大/
├── 1_1年前期/
│    ├── 細胞・遺伝学/
│    └── 情報処理学/
├── 2_1年後期/
│    ├── 生態・進化学/
│    └── 動物植物科学/
├── 2026_前期時間割.pdf
└── 2026_後期時間割.pdf
```

こんな感じで **4年生までを前期後期で分けて1～8までの数字を付けるのが良い感じな気がします。** なぜわざわざ番号を振るかと言うと、実は文字には番号が振られており(Unicode)、その順番だと **「前」** が **「後」** よりも番号が後ろにあるので、下のように **表示のときの順番が「後期」の方が上になって** めっちゃ気持ち悪いからです。<br>
まあ気持ち悪いだけですし、逆に数字があるほうが嫌な方は無しでもいいと思います。並び替えのルールを作成順とかにしたらなんとかなるとも思いますし。<br>
ちなみに、**もし数字をつけるなら絶対に1. 1のように間に空白を入れないで下さい。**
```
C:/Users/[ユーザー名]/Documents/農工大/
├── 1年後期/
│   ├── 生態・進化学/
│   └── 動物植物科学/
└── 1年前期/
    ├── 細胞・遺伝学/
    └── 情報処理学/

```

パソコン向けにもっと極端に作るなら全部英語にして
```
C:/Users/[ユーザー名]/Documents/tuat/
├── 1_fyear_fsem/ (fyear: 1年, fsem: first semester, 前期)
│    ├── cell・genestics/
│    └── information_processing/
├── 2_fyear_ssem/ (ssem: second semester, 後期)
│    ├── ecology_evolution/
│    └── animals_plalnts/
├── 2026_fsem_schedule.pdf
└── 2026_ssem_schedule.pdf
```
とはいえ、ここまでやるのは大変なので**農工大→tuat, 〇年〇期→〇year_〇sem** くらいに留めて、科目名は日本語のままにしとくのがちょうどいいと思います。

#### 4.1.3.1. フォルダやとかファイルの命名で意識すること
ということで、あんまり教えてもらえない ファイルやフォルダの命名するときに覚えておいた方がいいことを書いておきます。
- **空白の代わりにアンダースコア(`_`)を使え** : 空白はまじで使わない方がいいです。リボ払いと同じくらいやめた方がいいです。
`2026_01_ファイルの命名方法.docx`とかそういう感じが推奨です。Windows自体がC:に`Program Files`という空白付きのdirを作っているせいでまじでよく不具合が起きます。頭がおかしいです。

- アンダースコア`_`とハイフン`-`　: どっちがいいか分かりません。 **好みで良いと思います。** 僕はなんか気持ち的に`_`を使ってるんですが分かんないです。<br>
ちなみに、`.`で区切るのは拡張子と混ざるのでやめてください。

<details><summary>アンダースコアとハイフンは一応違いがある？</summary>
エクスプローラー上であるファイルをクリックして、0.5秒後くらいに撫でるようにクリックをすると、さらにその0.5秒後くらいに名前の変更のモードになります。(このラグがめっちゃ不愉快です)<br>
そのときに「`-`だと`-`と`-`の間の部分しか選択されなくて、`_`だと全部選択される」みたいな言説を見たんですが、手元のパソコンでやってみても「どっちも全部選択されてるやん」って感じで何も違いが分からないので、好みでいいと思います。<br>
また、URLはハイフンで区切るのが良いとされており、そのためGitHubのリポジトリもURLでもあるのでハイフンで区切りましょう。
</details
&#13;&#10;

&#13;&#10;
- **本当はできる限り英語が良い** : プログラミングとかをやりたい方は意識してほしいんですが、日本語は文字化けとかもありますし 入力時に **「Ener」で確定しないと入力したことにならない** ので、打ちながらリアルタイムでファイルを探したりする機能(`zsh`のtab補完, `fzf`, `zoxide`など)のときに英語に比べて明らかに使いづらいです。<br>

## 4.2. ショートカットキーを覚えよう
スマートフォンやタブレットとパソコンの違いはショートカットキーがあることです。ショートカットキーは無限大ににありますし、「これいつ使うねん」みたいなものも多いですし、[タイピングを簡単にしよう](#seven-zero-two)でも書いているんですが自分で設定して日本人向けに使いやすいように変えることもできます。<br>
つまり、沼なんです。沼なんですが、今日は超重要なものだけをピックアップして伝えます。

- **Ctrl + c : コピー。** copyのc。ターミナルではコマンドの強制終了になるので、ターミナルではShift + Ctrl + cでコピーします。
- **Ctrl + x : 切り取り**
- **Ctrl + v : 貼り付け**
- **Shift + Ctrl + v : 書式なしで貼り付け。** Wordとかで太字だったり見出し扱いになってるテキストを、そういう装飾を無くした ただのテキストとして貼ることができます。
- **Ctrl + z : 1つ前に戻る。** 操作を間違えたときなどに戻すことができます。
- **Ctrl + y : 1つ後ろに進む。** Ctrl+zの逆。
- **Ctrl + s : 保存。** saveのs
- **Ctrl + f : 検索。** findのf
- **Ctrl + N : 新しいウィンドウもしくは新しいタブを開く。** newのn。Google ChromeやMicrosoft Edgeなどのブラウザでは新しいウィンドウを、VSCodeやWezTermでは新しいタブを開く(そういうやつはShift + Ctrl + Nで新しいウィンドウを開く)。
- **Ctrl + T : 新しいタブを開く。** tabのt。
- **Ctrl + B : 太字にする。** boldのb
- **Shift + Enter : 改行**
- **F7 : カタカナに変換**
- **Win(四角4つのキー) + I : 設定アプリを開く**
- **Win + x + u + u : パソコンをシャットダウン。** パソコンが固まったようにときも、マウスが動かなくてもキーボードは普通に反応することが多い。キーボードだけでシャットダウンしておく方法を覚えておくと便利。
<br>

ちなみに、ショートカットキーが無限な理由は「アプリごとにショートカットキーが決まっているから」です。上に挙げたのはあくまで「一般的な挙動」であって、Ctrl + Kなどはアプリごとに動作がかなり違います。<br>
VSCodeというプログラミングによく用いられるソフトにはVSCode独自のショートカットキーの動作が決まっていますし、ターミナルなんてもっと一般的なソフトとは違ったショートカットキーをしています(Ctrl + cが分かりやすいですよね)。

# 5. パソコン基礎知識

<a id="five-one"></a>
## 5.1. OSの話をしよう ― OSは宇宙法則である。
パソコン講座は終わったのでちゃんと勉強っぽい内容に入っていきます。<br>
ということで、まずはOSというやつの話からしましょう。<br>
OSというのはOperating Systemの略です。まあ覚えてなくていいです。<br>
そして、おそらく多くの人はWindowsのパソコンを使っているのですがWindowsというのがOSです。MacBookを使っている人はMacOSが入っていて、おそらく入れてる人はいないと思いますがLinuxというOS(+の総称)もあります(Ubuntuとか僕が最近ハマってる好きなNixOSとかはLinuxです)。

そして、それぞれは一体何なのかというと、**OSは宇宙法則**だと思ってほしいです。OSというのが世界の在り方、例えば世界の構成だったり物理法則を規定しており、OSによって構成や物理法則が異なる みたいな感じだと思ってください。

とはいえ、私たち一般人は「このOSが使いやすい」とかそういうのはあっても「このOSはどんな法則が規定されてるんだ」的な内部のことはあまり考えなくていいです(構成は多少考えた方がいいんですが)。それを勉強するのは完全にコンピューターサイエンス(CS)系の勉強になります。

<a id="five-two"></a>
## 5.2. ディレクトリの話をしよう ― <code>C:</code>が宇宙, <code>~</code>が星

<a id="five-two-one"></a>
### 5.2.1. dir構造について ― ディレクトリ≒ フォルダ
では、プログラミングをやる上で絶対に外すことができない概念、ディレクトリについて説明していきます。まず、ディレクトリというのはいわゆるフォルダのことです。今後は省略してdirという表記をしていきます。

そして上で「OSごとに世界の構成が異なる」と書いたのですが、その世界の構成というのがディレクトリ構造というものだと思ってください。  
ディレクトリ構造というのは、「このディレクトリ(フォルダ)の下にはフォルダ Aとフォルダ Bとフォルダ Cがあって、フォルダ Aにはこのファイルが入ってる」みたいなことです。

具体例として、Windowsのホームディレクトリ直下のdir構造で説明します。<br>
ホームディレクトリ(home dir)というのは、OSという宇宙法則で作られている宇宙があるとしたときに、home dirが星の最高司令部みたいなものです。Windowsにおけるhome dirは以下のようになっています。
ぜひエクスプローラーというソフトで「PC → 謎の文字列+(C:) → Users → 自分のユーザー名(publicとかじゃないやつ)」とクリックしていけば下のと同じのが出てくるはずなので、クリックしながら「ほんまや」って理解していきましょう。

```text
/C:/Users/<ユーザー名>/ (= ~)
├─ Desktop
├─ Documents
├─ Downloads
├─ Music
├─ Pictures
├─ Videos
├─ AppData
│  ├─ Local
│  ├─ LocalLow
│  └─ Roaming
├─ OneDrive
├─ .local (最初の状態では存在しない)
│  └─ bin  
└─ (そのほかユーザーが作成したフォルダ)
```

そして、プログラミングをやっているとよく出ている`~`はhome dirのことです。  このあと`~`については説明します。
また、Windowsの宇宙が始まるところ(`C:`, Cドライブと言います)から見た一般的なdir構造は以下の通りです。

```text
/C:/
├─ Program Files
├─ Program Files (x86)
├─ Users
│  └─ <ユーザー名>  (= home dir)
│     ├─ Desktop
│     ├─ Documents
│     ├─ Downloads
│     ├─ Music
│     ├─ Pictures
│     ├─ Videos
│     ├─ AppData 
│     └─ .local (最初の状態では存在しない)
│        └─ bin
├─ Windows
├─ ProgramData
├─ PerfLogs
└─ (環境によって追加のフォルダ)
```

<a id="five-two-two"></a>
### 5.2.2. dirの表記 ― パソコン内の場所を記述できるようにしよう。
察しの良い人は気づいたかもしれませんが、上のdir構造って住所のようになっています。そして、住所であるならば住所を文字で表現することができるはずです。  

ここまでのdir構造を使って、場所(path)の書き方を整理します。  
Windowsでは `C:/Users/<ユーザー名>/Documents` のように、先頭から全部書くと絶対パスになります。
そして、dirやファイルの住所を表した文字列をパスと言います。  
(厳密にはパスは文字列ではなく、住所そのものなのですが、一旦これで説明します)

<a id="five-two-two-one"></a>
#### 5.2.2.1. 絶対パス ― 神の世界から見た住所
そして、パスの書き方には「絶対パス」と「相対パス」の2通りが存在します。実務上は相対パスの方が使いやすいのですが、dir構造を覚えるためにも絶対パスから説明します。

絶対パスというのは「神の世界から見た」パスです。神の世界というのは下の`C:`のさらに上の階層です。全ての宇宙を見渡せる階層とでも言えばいいのでしょうか？詳しくは下の [補足: なぜ最初に <code>/</code> を付けるのか。](#why-leading-slash)を参照してください。

例えば、
```
C:/
├─ Windows
├─ ProgramData
└─ [その他色々]
```
に対して、`ProgramData`というdir(フォルダ)は`C:/ProgramData/`とか`/c/ProgramData/`とかって書きます。  
Windowsの宇宙法則では`C:\ProgramData\`のように`\`(バックスラッシュ)で書くように内部でなっているのですが、MacやLinuxをはじめとして、基本的にプログラミングの世界では`/c/ProgramData/`のような表記をすることが多く、Windowsでもこの書き方は許される(PowerShellはよく分かりません)のでこの書き方をしましょう。

なお、`/c/ProgramData/`と`/c/ProgramData`のように、最後に`/`をつけてもつけなくても大丈夫です。僕はつけたりつけなかったりですが、最初の方は「これはdirなんだ」ということを明示するために`/`をつけておくと良いような気はします。
<details id="why-leading-slash">
  <summary>補足: なぜ最初に <code>/</code> を付けるのか。</summary>
    まず、OSは宇宙という話をしましたが、パソコンは様々な宇宙を内包させることができます。MacOSはApple製品にしか載っていないだけで、パソコンに複数のOSを導入することは可能なんです。よくあるのはWindowsパソコンにLinuxを載せるやつです(WSLというWindowsの宇宙に仮想のLinuxの宇宙を創るやつとは違います)。そして、<code>/c</code>というのは、パソコンと言う神の世界から<code>(神の世界)/c</code>という特定の宇宙の世界(この例ではWindowsの宇宙)にアクセスしていることを意味していると捉えることができます。
</details>
&#13;&#10;  

&#13;&#10;

次にhome dir配下のパスの話をします。  
まず、下の図を見れば分かる通りhome dirは`/c/Users/[ユーザー名]/`というパスになっていますよね。  例えば、ダウンロードディレクトリ(フォルダ)は`/c/Users/[ユーザー名]/Downloads`となります。  
ダウンロードディレクトリに`tuat_daisuki.md`というファイルがあるならば、このファイルのパスは`/c/Users/[ユーザー名]/Downloads/tuat_daisuki.md` になりますよね。

ちなみに、エクスプローラー(黄色と水色のアプリ)では「ダウンロード」のように日本語で書かれていますが、内部的には英語で登録されているので英語で書くようにしましょう。

```
/C:/
├─ Program Files
├─ Program Files (x86)
├─ Users
│  └─ <ユーザー名>  (= home dir)
│     ├─ Desktop
│     ├─ Documents
│     ├─ Downloads
│     ├─ Music
│     ├─ Pictures
│     ├─ Videos
│     ├─ AppData
│     └─ .local (最初の状態では存在しない)
│        └─ bin
├─ Windows
├─ ProgramData
├─ PerfLogs
└─ (環境によって追加のフォルダ)
```

<a id="five-two-two-two"></a>
#### 5.2.2.2. 展開パス ― `~`が便利！！
そして、home dirには特殊な性質があります。それが、`~`というホームチルダと僕が呼称しているやつです。というのも、そもそもプログラミングにおいては 必要なファイルなどは全てhome dir配下、つまり星の中で管理するのが美しいとされています。宇宙に置くのはあまり推奨されません。  
そういう運用を心がけると、ユーザーはhome dir以降での操作が多くなりますよね。それを毎回真面目に書くと`/c/Users/[ユーザー名]/Documents/`などのようになってしまいとっても煩わしいですよね。また、他のパソコンを触るときにいちいちそのパソコンのユーザー名に合わせて打つというのは面倒くさいですし再現性が低いですよね。

そのため、`/c/Users/[ユーザー名]/`の部分を`~`で代替することができます。
つまり、`~/Downloads/test.txt` は `C:/Users/<ユーザー名>/Downloads/test.txt` と同じです。

<details id="home-dir-handling"><summary>パスを扱うプログラムの<code>~</code>の扱い</summary>
今までした<code>~</code>の話はターミナル(シェル)の話であって、パスを使ってファイルを動かしたりコピーしたりするプログラムを自分で作った場合には自動で`~`が展開されるわけではないので、「`~`があったらホームチルダとして展開しようね」みたいな設定が必要になります。
</details>

<a id="five-two-two-two-one"></a>
##### 5.2.2.2.1. デフォルトの`cd`, `z`の挙動 ― home dirに移動する。
ちなみに、home dir配下(=より下の階層)がパソコンの世界でよく用いることを表す事実として、dirを移動するのに用いる標準コマンドの`cd`や、僕が大好きな外部導入コマンドの`z`(`zoxide`)が引数を取らないときにhome dirに移動するというのがあります。

この後コマンドについての解説は行うのですが、`cd`について簡単に動作を説明します。(`z`は`cd`の上位機能版なので`cd`で説明します)
`cd`はchange dirの略で、このコマンドを使うと、例えば
```
hirose-suguur@everyday-happy ~
$ mv Documents/農工大/1年前期/情報処理学/test.txt Documents/農工大/1年前期/細胞・遺伝学/
```
(`mv`はdirやファイルを動かすコマンドです。`情報処理学/`にあるファイルを`細胞・遺伝学/`に動かしています。)  
とするところを
```
hirose-suguur@everyday-happy ~
$ cd Documents/農工大/1年前期/

hirose-suguru@everyday-happy ~/Documents/農工大/1年前期/ ←これが今いるdirを表しています。
$ mv 情報処理学/test.txt 細胞・遺伝学/
```
とするだけでよくなります。簡単に言うと、ある程度同じdirまで進んだあとの分岐するような動作をするときに、先に`cd`を使うと入力が楽になります。

そして、その`cd`は
```
hirose-suguur@everyday-happy ~
$ cd Documents/農工大/1年前期/

hirose-suguru@everyday-happy ~/Documents/農工大/1年前期 
$ cd

hirose-suguru@everyday-happy ~
$
```
のように2つ目のコマンドのようにどこかのdirに移動した後で`cd`の後ろに何もつけずに`cd`をするとhome dirに移動します。これはhome dirより上の階層(宇宙の方)にいても同じです。

<a id="five-two-two-three"></a>
#### 5.2.2.3. 相対パスについて ― 実務的にはこっちを使うことが多い。
先ほどまでは 宇宙を超えた神の世界を基準にパスを書く 絶対パスを説明しましたが、次は今いる場所を基準にした相対パスの説明をします。先ほどから述べている通り実際にプログラミング的なことをするときには相対パスの方が遥かによく用いられます。

まず、実は既に相対パスを用いています。それが下の例です。
```
hirose-suguur@everyday-happy ~
$ cd Documents/農工大/1年前期/

hirose-suguru@everyday-happy ~/Documents/農工大/1年前期
$ mv 情報処理学/test.txt 細胞・遺伝学/
```
この2つ目のコマンドは`~/Documents/農工大/1年前期`を基準としているため`情報処理学/`や`細胞・遺伝学/`にパパっとアクセスできています。これは`~/Documents/農工大/1年前期/`を基準とした相対パスで指定しているわけです。

また、`cd`が1階層浅いならば
```
hirose-suguur@everyday-happy ~
$ cd Documents/農工大/

hirose-suguru@everyday-happy ~/Documents/農工大
$ mv 1年前期/情報処理学/test.txt 1年前期/細胞・遺伝学/test.txt
```
のようになります。

そして、ここまでは直感に即しているので分かりやすいと思いますが、2つだけ相対パスの独特の記法があります。それが以下の2つです。
- `./` : 今いるdir (current dir, カレントdir)。
- `../` : 1つ上のdir

(どちらも`/`が無くてもいいですが`..`を連続して使うときは`../..`のように1回ごとに`/`を入れないといけません)

これを駆使することで全てのdirを相対パスで表せるようになります。
例えば、今`~/Documents/農工大/`にいるとして、  
`~/Pictures/スクリーンショット/スクリーンショット(1)`は  
`..(これでDocuments)/..(これでhomedir)/Pictures/スクリーンショット/スクリーンショット(1)`のように表せます。

また、今いるdir (current dirと言います)は何に使うのかと思うかもしれませんが、`cd`などで移動するときに用いるのではなく、その他のコマンドで用いることが多いです。
例えば、`git`というシステムを用いる場合にステージングという操作があるのですが、そのときに
```
git add .
```
という操作をよくします。

## 6.1. 環境と開発環境の違い ― 開発環境の方が広い概念
ということで、プログラミングの1大テーマ, 環境構築の話をします。そして、本来は環境構築という言葉の意味をバチっと話したいのですが、ITの世界では「環境」と「開発環境」の2つの用語があり、それぞれを区別した方が良いのでそこから話します。なお、**開発環境は「本番環境」の対置の意味合いで使われることも多いですが、今回はそれは省きます。**

- 環境: ある言語やツールを用いることができる(入っている)状況
- 開発環境: 何かしらのプログラミングなり何なりをする(開発)ために入れた全てのツールを総合した名称。環境を含んだ概念。

例えば、これからみなさんはPythonを勉強しながらコードを書きますが、これを英作文に置き換えてみましょう。英作文で言えば
- 環境: 英語という概念, 目, 手, 英語を勉強する部屋
- 開発環境: 英語という概念, 目, 手, 英語を勉強する部屋, 丸付けしてくれる先生, ペン, ノート, 辞書

みたいな感じです。

もちろん喩えなので厳密に対応しているわけではないですが 簡単に言うと、環境は「それを行うのに厳密に必要なもの」、開発環境は「それを行うために用いている色々なもの」をまとめて開発環境と言うわけです。環境も「厳密に」とはありますが割と曖昧ですし、開発環境はもっと曖昧です。プログラミングの調べ物をするなら「Chromeも開発環境に入る！！」と言い張ることはできるわけです。

で、なぜここで「開発環境」の話をするかと言うと、この農学部の中で数少ないプログラミングをちゃんとやってみたい、興味がある人は、初めから優れたペンやノート, 辞書を持っていた方が英作文がしやすいですよね。そして、もうこれまでの先人が優れた環境構築の知見を積み上げてきているので、とりあえず脳死で模倣した方が良いです。Vaundyもまずは模倣からって言ってました。

**ちなみに、この後話す開発環境構築はWindows向けなのでMacOSの人は自分で頑張って下さい。**

#### 7.0.4.1. ターミナルとシェルの違い ― コマンドプロンプトもPowerShellもはターミナル言語
Windowsのやばい設計のかなり上位の方に入ってくるものとして、「コマンドプロンプト」とか「PowerShell」とか「Windowsターミナル」の区別がつきにくいってことです。

そして、そのコマンドの書き方がシェルごとに決まっており、そこで差別化が行われます。

一方で、ターミナル(+アプリ)というのはそれぞれのターミナル言語を使える器みたいなものです。厳密には違いますが、パソコンはパソコンがあれば日本語も英語も中国語も他の言語も打てますよね。これでいうパソコンがターミナルアプリです。Windowsに標準で入ってるターミナルアプリがWinows Terminal, MacOSはiTerm2, 後から自分で入れるやつだと僕が使ってるWezTerm, 最近流行ってるGhossty, MacOSでしか入れれないcmux, ちょっと古いのだとtmuxなどがあります。

まずは、このシェルとターミナルアプリの違いを理解しましょう。

#### 7.0.5.1. IDEとテキストエディタの違い ― VSCodeはIDE

ここで、VSCodeとかいうソフトが圧倒的に有名かつ安定択すぎてVSCode以外のテキストエディタやIDEを使わないためにVSCodeが一体何なのかを理解していないことが多いので、ここでVSCodeを使ってテキストエディタやIDEの違いを話しておきます。

先ほども述べたようにVSCodeはIDE(Integrated Development Environment, 統合開発環境)です。一方で、VSCodeはよく「テキストエディタ」というように説明されます。この表現は「そりゃ正方形は平行四辺形だけど正方形に平行四辺形って言わなくない？」みたいな気持ち悪さがあります。

というのも、先ほど僕がちょろっと使っていた<br>
「超絶多機能なメモ帳をさらに魔改造させたもの」というのは

- **めっちゃ使いやすいメモ帳** ← テキストエディタ
- **↑を魔改造させたもの** ← IDE(統合開発環境)

という包含関係になっています。つまり、テキストエディタはめっちゃ便利だとしてもあくまでメモ帳で、IDEはテキストエディタを魔改造させて「これはもうメモ帳・・・なのか？」ってなってるやつなので、IDEは確かにテキストエディタではあるんです。

そして、じゃあ「どう魔改造したのか？」という話になりますが、これは開発環境の意味から考えると分かりやすいです。<br>
開発環境は英語で言う<br>
「英語という概念, 目, 手, 英語を勉強する部屋, 丸付けしてくれる先生, ペン, ノート, 辞書」であり、意味合いとしては「それを行うためにその人が使っていると思っている色々なもの」でしたね。

つまり、どう魔改造されたかと言うと、「めっちゃ使いやすいメモ帳に色んな機能がついて単独で開発環境になってしまった」という点で魔改造されたんです。
「英語という概念」以外の全てが入っているまであります。
まず、
- 目: コードを読み取ってVSCodeの中で結果を出力できる(実行環境)
- 英語を勉強する部屋: ワークスペース
- ノート: コードが書ける
- 丸付けしてくれる先生: コードを解析して「ここミスってね？」ってリアルタイムで教えてくれる
- 辞書: プログラミングの関数の使い方とかを表示してくれる

などです。「英語という概念」に関しては、パソコン内にPythonを入れないとVSCodeがあってもPythonのスクリプト(`.py`)を「目」が読み取って実行してくれません。しかし、VSCodeはGoogle Colabと連携できるので、そういう意味では「英語という概念」すらも疑似的に手に入れていると言えるかもしれません。流石に魔改造すぎます。

# 8. プログラミング言語概論
ということで、環境構築は書ききれないのでまずはプログラミングの理論の方を書いていきます。<br>
まずは、簡単かつ典型的な書き方のpythonのファイルを見て、どのような要素でpythonのファイルが構成されているかを見てみましょう。あと、今後はですます調がめんどくさいのでである調に変わります。

<a id="eight-one"></a>
## 8.1. プログラミング言語の種類と違い
プログラミング言語は数多く存在する。例えば、Python、Rust、Kotlin、TypeScript、Go、あるいは関数型と呼ばれるHaskellなどがある。これらは大きく2つの軸で分類できる。**コンパイルするかしないか**と、**型を宣言するかしないか**である。

<a id="eight-one-one"></a>
### 8.1.1. コンパイル言語とインタープリタ言語
コンパイルとは、人間が書いたソースコードをCPUが直接実行できる機械語に変換する作業のことである。機械語というのは「0と1のみで記述されたコンピューターの共通言語」のようなものであり、機械語で書かれたファイルを **バイナリファイル** という。
電子的なコンピューターが発明されたときには機械語のみが存在していたため、全てのプログラミング言語は人類が意図的にバベルの塔で言語を散らばらせてできた産物という風に言えるかもしれない。

そして、**プログラミング言語から機械語への変換(コンパイル, バイナリ化)を事前に行う** 言語をコンパイル言語という。このコンパイルにより生成される機械語で書かれたファイルをバイナリ(+ファイル)と言う。<br>
一方で、**変換を行わず1行ずつ読みながら実行する** 言語を インタープリタ言語 という。

<a id="eight-one-two"></a>
### 8.1.2. 静的型付けと動的型付け
コンパイル ↔ インタープリタの対比と同じくらい重要な概念、というよりもコンパイルとインタープリタのそれぞれの言語設計の基盤となる概念が静的型付けと動的型付けである。コンパイル言語は静的型付けを、インタープリタ言語は動的型付け言語であることがほとんどである。

静的型付けというのは**コードの段階で「この変数の型(Pythonの解説で説明)」というのを宣言する(これを型宣言という)** ということであり、動的型付けは実行時にその変数の型が確定する。([クラスと型とクラス構造体](#one-zero-two))<br>
静的型付けによりコードの段階で型を決めていることで実行せずともコンパイルの段階でエラーが出る。

プログラミング言語を運用に近い観点で分類すると「コンパイル言語」「インタープリタ言語」であり、それを支える根源的な仕組みに着目すると「静的型付け言語」「動的型付け言語」に言い換えられる。

<a id="eight-one-three"></a>
### 8.1.3. コンパイル言語・静的型付け言語と、インタープリタ言語・動的型付け言語の利点と欠点
一般的に、コンパイル言語かつ静的型付け言語は、実行前にコンパイルが走るため、型の不一致や変数の未定義などの多くのエラーを実行前に検出できるため安全性が高い。また、コンピューターが読み取る最も根源的な(低レベルな)言語である機械語に変換済みであるため実行速度が速い。<br>
一方で、開発の段階で型宣言が必須であり、また変更したファイルを実行するためには毎回ビルドが必要な点でも開発速度は遅くなる傾向がある。

対して、インタープリタ言語かつ動的型付け言語は、型宣言が不要であり、コードを素早く書いて実行できるため、開発が簡単である。しかし、エラーは実行時に初めて発覚することが多く、実行速度も遅い。

具体例として、Rustはコンパイル言語かつ静的型付け言語の代表例であり、Pythonはインタープリタ言語かつ動的型付け言語である。<br>
Rustはメモリ安全性と実行速度が非常に優れていいるため、近年人気が高まっている。一方でPythonは開発が容易であるため初心者から企業までの広い技能レベル、そして広い領域で採用されている。

ただし、Pythonは型アノテーション([簡単な例](#one-zero-one))を書き、それに型チェックを適用することで実行前に型のチェックができるため、静的型付けの強みを取り入れるような運用もできる。

<a id="eight-one-four"></a>
### 8.1.4. AI時代のプログラミング言語
近年、AIがコード生成を行うようになったことで、プログラミング言語の選択基準が変わりつつある。<br>
静的型付けでコンパイルが必要なRustは、他の言語にはない概念がありそれが学習難度に繋がっていた。しかし、生成したコードのエラーをビルド時に即座に検出できるためAIのコードに対しての認知負荷が軽減され、またビルドもAIが自動で行えるため開発速度も制約がなくなっている。このため、AIコーディングの普及によりでますますRustの人気が高まっている。

一方、Pythonは過去数十年のコード資産が膨大にあり、AIの学習データが豊富である。そのため、言語仕様自体にAIとの親和性は特にないが、それをコード資源で補って、現時点ではPythonもAIが書きやすい言語になっている。

そして、人間が読む機会が減るなら今後は「人間が読まなくて安定した品質でプログラミングができるような仕組みづくりを行う」という部分に比重がシフトしていきます。

### 9.1.3. 今AIを使っている人たちが優れているのはプログラミングの知識ではない

技術は常に変わるから勉強し続けないといけませんし、そうでなくても日常生活の一部として新しい技術の勉強を平然とやっている人が沢山います。

AIがプログラミングを奪う可能性は否定しませんが、今最もAIを上手く使っている人たちは「1円も稼げない何物でもない状態」からプログラミングを学んできたような主体性と学習の継続力のバケモノです。そういう意味では「今プログラミングを勉強して何になるんだ」と考えている人と同じ立場だったとも言えます。

つまり、プログラミングどうこうではなく、「お金になるか分からなくても「そんなの関係ねえ」と主体的に継続して学び続ける能力」というのがAIの活用で大事なんだと思います。

<a id="nine-two"></a>
## 9.2. AIによるプログラミングで意識するべきこと ― 提供, 隔離, 制限
ということで、AIを活用したプログラミングの話をしていきます。
コラムのところにも書きましたが、AIによる開発は「提供」, 「隔離」, 「制限」の3つが大事だと感じているのでそのことについて説明していきます。
これらは完全に分かれるものではないですが、説明のために3つに分けていると思ってください。

<a id="nine-two-one"></a>
### 9.2.1. 提供 ― 「何を使えばいいのか」「何をすればいいのか」を与える

<a id="nine-two-one-one"></a>
#### 9.2.1.1. 何をすればいいかを与える

まず、Claude CodeでもCodexでもGemini-CLIでもいいのですが、CUIのエージェントを入れて使ってみると分かるんですけど、「AIはこのdirに存在しているファイルすら分かっていない」んです。つまり、AIは一般的な知識は豊富ですが、この場所の状況はAIに与えたり、AIが調べないと何もわかってくれません。

「このファイルにある関数をちょっと変えただけのことをまた0から実装してない！？」みたいな、「車輪の再発明」と呼ばれることを平気でやります。

それを防ぐには「こういう風にやろう」と言うことですが、毎回毎回プロンプト(人間が投げるメッセージです)に「こういう風にやればいいんだよ」と何百文字も書くのは非効率です。それならば、「ここではこうすればいいよ」というのを書いたファイルを読ませれば再現性良くAIの運用ができるわけです。
一般的な「運用ルールブック」はClaude Codeでは`CLAUDE.md`, CodexとGeminl-CLIでは`AGENTS.md`になっています。

また、「このファイルを読んで」と言うのも面倒なので、それもhookという機能を用いれば自動化することもできます。


<a id="nine-two-one-two"></a>
#### 9.2.1.2. 検証を与える

これはCUIのエージェントを使った人なら分かると思うのですが、AIのエージェントは時間の制約も納期もないのにすぐにコードを変更しようとします。「絶対このエラーはそういう問題じゃないけどな」みたいなくだらない細かい変更をして「修正しました」とかってほざきます。<br>
それでも「検証を大事に」というのを伝える(+仕組みを作る)こと、そして「検証はこうやってやろう」というのを整えていくことでCUIのエージェントは「ちゃんと検証をしてからコードを変えよう」という感じにやってくれます。

<a id="nine-two-one-three"></a>
#### 9.2.1.3. 応用編？ ― 段階的な情報の提供
生成AIをちゃんと追っている人は知っているかもしれませんが10か月前とかはMCPというのが流行っていました。簡単に言うと、誰かが作った拡張ツールを使えるようにするやつです。<br>
しかし、MCPは「このMCPはこういう機能があって、これが返ってきて」みたいなツールの知識を全て与えることになるのでトークン(計算量みたいなものだと思ってください)の消費量が激しいです。

そして、現在の生成AIのこれまでとの最も大きな差は「自律的な判断ができるレベルが非常に上がった」ことだと感じています。<br>
今までの生成AIは「学習したデータで得た知識を出す」という部分がメインで、未知のものに対してその場で理解と判断をすることがほとんどできていませんでした。

現代の生成AIはその場での理解と自律的な判断がかなりできるようになったため「とりあえずこういうときにこれを読んで」という入り口だけ与える段階的な情報の提供へとシフトしています。

簡単に言うと電話のコールセンターみたいなものです「○○したい方は1を」とかって言ってきて、私たち人間がそれを判断してボタンを押すじゃないですか。それをAIにもやらせるということです。

だからこそ「Aなときにこれを読んで、これを読んだら詳しいことは書いてあるから」という情報だけを与えれば「今はAじゃないから読まなくていいな」とAIが判断するのでトークンの節約になります。これがClaude Codeで言う、`.claude/skills/スキル名/SKILL.md`で記述されるSkillという機能です。

同様に、CUIエージェントはコマンドの使用もできるので「こういうときにこのコマンドを叩いて」というCLIとその情報だけ与えて、そのコマンドの結果を受け取ってその次の動作に繋げていくような運用も最近流行っており、MCPはこのCLI + Skillに変化してきています。

ちなみに僕が考えているのは自作のCLIの作成です。`.py`そこの`print`関数で教えたい情報を出力するようにして、`.py`をClaude Code自身が実行してClaude Codeが情報を能動的に得る方法です。<br>
原理的には普通のCLIツールと同じなので実運用もちょっとできているんですが、開発の仕方全体に組み込むのははまだできていないので、それをできるようにしたいと考えています。

### 隔離 ― セキュリティ的隔離と情報的隔離

#### セキュリティ的隔離
これは僕もあまり詳しくない領域です。CUIのツールはファイルにアクセスでき、コマンドも叩けるからこそ そのパソコンにあるデータを全て吹き飛ばすことも可能です。<br>
だからこそ、「どこで作業させるか」「何だけさせるか」というのを意識するのが大事です。いわゆるコンテナという隔離環境を用意することはセキュリティ的隔離の代表例です。

#### 情報的隔離
僕がそれなりに実践できているのはこれです。Claude Codeに悪意はありませんが、何も教えてないとてきとーなことをしますし、また書いてあることをとりあえず遵守する傾向が高いです。<br>
その性質を上手く使えば「こういうときはこうしろ」というのをきちんと与えておけばやってほしくないことをする可能性は低くなります。<br>
「汎用的なツールはここに書いてあるから最初に見て、あとファイルはここに作って」という提供を与えておけば、枝葉の関係ないファイルの実装を見てそれに従って車輪の再発明をすることはありません。これが情報的隔離です。

###### コンテキスト汚染
情報的隔離は致命的なミスをClaude Codeがしないためでもありますが、他の利点もあります。それがコンテキスト汚染を防げるという点です。

例えば、あるファイルを編集したいときにそれが見つからないとしたときに、Claude Codeが15回とかコマンドを叩いて探して見つけたら、そのときのコマンドの結果全てをClaude Codeが読み取ってしまいます。
その後の開発には他のファイルの情報なんて全く必要ないですよね。人間には基本的に「必要ない情報を捨て去る」能力が高い(多分)ですが、AIはそれが不得意なので関係ない情報も記憶して、それが今後のトークンの使用量などに悪影響を及ぼす可能性があります。これをコンテキスト汚染といいます。(生成AI移行にできた言葉なので定義は定まってませんが。)

#### SubAgent
SubAgentはClaude Code自体が提供してる情報的隔離の最も代表的な手法です。
Skillが「手法の段階的提供」であるのに対して、SubAgentは「手法の段階的提供と隔離」の両方を行えるものです。

例えば、ボードゲームみたいなものをプログラミングで作ったとしましょう。そして、それのルールがどうかを確かめるときに実際にClaude Codeにあそばせてみるとしましょう。そうすると「実際にゲームを遊ぶときはこのSubAgentを立ち上げて」ということと、そのゲームのルールを書いたファイルを`.claude/agents/エージェント名.md`に書いておけば<br>

Claude Codeメイン人格「よーしゲームをするぞー。お、このファイルにはゲームをするときに立ち上げろって言ってる、よーし起動！！」
Claude Code第二人格「ふむふむ、私は実際にゲームをすればいいわけだな。ほう、ルールはこうなっているのか。あと、私はファイルの書き込みができないのだな。ではやってみよう。」→「こんな感じか。このルールはこう改善した方がいいだろう。おいっす、メイン人格さん、なんかルールなんすけど、こういう風にした方がいいかもっす。」
Claude Codeメイン人格「こういう風に変えた方がいいかもです。」

のように、Claude Codeが別の人格的なものを呼び出して、特定のタスクだけをやらせる感じの運用ができます。このとき、メインのエージェントはゲームをプレイしたときの細かいことは全くコンテキストに入ってきません。<br>
サブの人格さんに関してはゲームを実際に遊ぶことでのコンテキストの汚染を全て引き受けることができますし、また「俺はファイルの書き込みとかはできない」とかの制限も与えることができます。ファイルの読み込みすらも禁止してただただコマンドを打ち続けるだけのサブ人格さんを作ることができます。

つまり、ある程度のセキュリティ的隔離とかなりの情報的隔離を行えているということです。

### 制限

これは割と頑張っています。簡単に言うと、「良くない操作をしようとしてる」というのを検知してブロックする仕組み作りです。どれだけClaude Codeに情報を与えても、TransformerというAIの仕組み上 記憶は失われていくようですし、また「これをするな」というのを全て書くことはトークン的にも厳しいです。だからこそ、**「するな」という自然言語ではなく、システムとして制限すること** が大事です。それの主な手法が`settigns.json`の`permissions`の設定だったり、hookによるブロックです。<br>

hookというのは「特定の動作に反応して特定のプログラムを走らせる」機能です。例えば、Claude Codeを起動したときに自動で「まずはこのdirのファイルを読んで」というメッセージを出すことができます。

そして、Claude CodeはToolの使用前にもhookを仕組むことができるので、「もしこういうことをしようとしてたら終了コード2を出してブロック」というような設定も可能なわけです。

hookによるブロックは非常に奥が深く、まだまだ僕自身が上手く扱いきれていないので、僕の今後の課題でもあります。

# 10. Python基礎(+理論)

<a id="one-zero-one"></a>
## 10.1. 簡単な例
``` python
# example.pyに書かれているとします。
def function(hop: int, step: int, jump: int) -> int:
  """3つの整数を足し合わせて返す関数"""
  fly_high = hop + step + jump
  return fly_high

if __name__ == (__main__)
  the_farthest_future = function(1, 2, 3)
  print(the_farthest_future)
 
#これを実行したときの標準出力
6
```

- 上のコードにおいて、`def function`から`return fly_high`までが関数の定義部分である。<br>
たとえば、パン工場で言えば、様々なパンが流れてきてそのパンに応じて袋詰めをして製品として出荷する。その袋詰めの仕組みに当たるのが関数の定義の部分である。

- また、`function(hop, step, jump)`における`hop, step, jump`を引数(ひきすう)という。パン工場では流れてくるパンに対応した袋で包装するが、その時々の流れてくるパンが引数である。

- `return fly_high`では関数の結果として`fly_high`という値が出力される。パン工場でいう袋詰めされた後の商品である。<br>
今回は整数が出力されるが、このドキュメントを通して **「値」は数値を意味しない。** また、今回のコードを含めてはじめは`return ○○`の形で値を返すことが多いが、返り値を返さないような関数もある。

- 引数である`hop`, `step`, `jump`と関数内で定義された`fly_high`をまとめて「変数(variable)」という。特に、関数の部分で定義された変数であるため「ローカル変数」という。

`def function(hop: int, step: int, jump: int) -> int:`における`hop: int`と(`step`, `jump`も同様)`-> int`の部分を型アノテーション(annotation)という。これは「`hop`という引数は`int`型である」「この関数の返り値は`int`型である」と引数の型と返り値の型を宣言している。

型アノテーションがあってもPythonは動的型付け言語であるため型アノテーションは実行に影響しない(つまり、引数に型アノテーションと異なる型が与えられたりしても即座にエラーにはならない)が、コードを読む人間のコードの可読性を担保するほか、型チェッカーは型アノテーションをもとにエラーを検出することができる。

また、`"""3つの整数を足し合わせて返す関数"""`という部分をdocstring(ドキュメンテーション文字列)という。これは関数が何をするかを説明する文字列であり、`help(function)`で参照できたり、IDEの補完機能で表示されたりする。

`if __name__ == (__main__)`は実行のための呪文のようなもので、単体でこの`.py`ファイルを実行したときにのみ実行される内容が下に記述されている。これによりテストのしやすい設計を担保している。<br>
仮に`if __name__==(__main__)`が無い場合はこの`.py`を`import`して用いた場合にも
```
the_farthest_future = function(1, 2, 3)
print(the_farthest_future)
```
の部分が実行されてしまう。

## 10.2. クラスと型とクラス構造体

### 10.2.1. 型という概念
プログラミングにおいて、値はすべて何らかの型(type)を持つ。 **型とは「その値がどのような種類のデータか」を示すラベル** であり、かつ型に応じてその型が持つ「属性(attribution)」も規定される([属性(attribute)という概念](#one-zero-three-three))。

よく使う型の一覧を以下に示す。

| 型 | 意味・内容 |
|---|---|
| `str` | 文字列。stringの略。`"hello"`, `'world'`など |
| `int` | 整数。integerの略。`1`, `-5`, `100`など |
| `float` | 浮動小数点数。floating-point numberの略。小数点が浮動する数。`3.14`, `-0.5`など |
| `bool` | 論理値型。booleanの略。`True`, `False`の2つの値のみが`bool`型である。 |
| `list` | 順序付きの可変長配列。`[1, 2, 3]`など |
| `tuple` | 順序付きの不変配列。`(1, 2, 3)`など |
| `dict` | キーと値のペア(連想配列)。`{"key": value}`など |
| `NoneType` | `None`のみを値として持つ型。値がないことを表す。 |

下のは少しマニアックな型である。

| `set` | (重複無しの)集合。`{1, 2, 3}`など |
| `frozenset` | `set`の不変版。変更不可 |
| `complex` | 複素数。`1+2j`など |
| `bytes` | バイト列。バイナリデータを扱う。`b"hello"`など |

<a id="one-zero-two-two"></a>
### 10.2.2. インスタンスと型
Pythonは動的型付け言語という話はしたが、それが具体的に何を意味するかというと **「インスタンス(値)が先に存在して、それに対して名前(変数名)をつけている」** ことである。
インスタンスというのはその型を持った実際の値(値$\neq$ 数)のことである。

例えば、以下のコードを考える。
```python
a = "hello"
```

初心者の人は「`a`は`"hello"`である」、少し知識がつくと「変数 `a` に文字列 `"hello"` を代入した」と考えるが、Pythonでは異なる。<br>
実際には

1. `str`型の`"hello"`というインスタンス(事物)がメモリ上に存在する。
2. `a = "hello"`で、その`"hello"`というインスタンスに`a`という名前をつけた。

というようになっている。

つまり、変数 `a` は値そのものではなく、値(`"hello"`)に対してつけた「名前」に過ぎない。<br>
比喩的に捉えるなら、式の右辺はインスタンスを世界に顕現させ、左辺はそれに名前をつけていると言える。

このため、Pythonでは同じ変数に異なる型の値を再代入できる。
```python
a = "hello" # str型のインスタンス"hello"を参照
a = 10      # int型のインスタンス10を参照
a = [1, 2]  # list型のインスタンス[1, 2]を参照
```

また、Pythonは **本質的には「この変数の型はこれ」と決まることがなく、あくまで「このときのこの変数(+が参照しているインスタンスの型)はこの型」というその瞬間の型が決まっている** のみである。これが動的型付けの意味である。

対して、Rustなどの静的型付け言語では、変数の型が宣言時に決まり、実行中に変わらない：
```rust
let a: &str = "hello";  // aの型は&str（文字列型）と宣言
// a = 10;              // エラー：異なる型は代入できない
```

### 10.2.3. クラスと構造体
クラスとは、「ある変数がどのような型で, どのように構成されているかを定義したもの」である。`str`もある変数が1つの`""`の形になっていると定義されるため、クラスである。

そして、Pythonでは`class A:`でクラスを新たに定義できる。このように定義されたものをクラスと呼称することが多いが、`str`も`int`も`list`も`tuple`も新たに定義したクラスではないがクラスではある。<br>
よって、「クラス」という名称での混乱を避けるために`str`, `int`, `list`, `tuple`などの**ユーザーが新たに定義したわけではないクラス**に関しては**型**と呼ぶことにする。

そして、`class A:`で新たに定義したクラスを(このドキュメント内では)**クラス構造体**と呼ぶことにする。他の言語における構造体(structure)は複数のデータ属性のみを持つクラスであり、Pythonの`class A:`で定義されたクラスはメソッドも持てるため、区別のためにクラス構造体と呼称する。<br>

詳しくはデータ属性, メソッド属性の説明 → [属性(attribute)という概念](#one-zero-three-three)を見てもらいたいが、一旦クラス構造体と構造体のイメージがつくように具体例で考えてみる。<br>
例えば下の`Student`は、データ属性(`id`, `name`, `year_level`)とメソッド属性(`introduce`)を持つため、`Student`クラスはクラス構造体である。
```python
class Student:
    def __init__(self, name, score): #一旦理解せず無視してOK
        # self.〇〇でStudentクラスの持つデータ属性を定義
        self.id = id                  # データ属性
        self.name = name              # データ属性
        self.year_level = year_level  # データ属性

    def introduce(self):      # メソッド属性
        return f"My name is {self.name}."
```
もし`id`や`name`のようなデータ属性だけを保持し、メソッドを持たないなら、他言語の意味での構造体に近い。

例えば、Rustにはクラス構造体はなく単なる構造体しかない。次のように`struct`でデータ属性のみを定義する。<br>
メソッド属性も定義したい場合は構造体とは別で`impl クラス名`で別に書く。
```rust
struct Student {
    id: u32,  // 0 ～ 4,294,967,295の非負整数
    name: String,
    year_level: u8, // 0 ～ 255の非負整数
}

impl Student {
    fn introduce(&self) -> String { // メソッド属性
        format!("My name is {}.", self.name) // これがreturnにもなる
    }
}
```

なお、「オブジェクト」という用語があるが、オブジェクトはインスタンスを意味することもあれば(クラス)構造体を意味することもあり、意味が混同される可能性が高いためこのドキュメントでは用いない。

### 10.2.4. 主要な型

<a id="one-zero-two-four-one"></a>
#### 10.2.4.1. `str`
stringの略もっとも身近な型の1つ。`"hello"`などの`""`や`''`の囲み文字を含めた文字列のことを表す。<br>
下のコードのように`+`演算子で文字列の長さを変えられていることから分かるように、`str`も可変長配列であり、イテラブルな型である。([配列とイテラブル](#one-zero-two-five))

```python
example = "hello"
example = example + " world"
print(example)  # "hello world"
```

<a id="one-zero-two-four-two"></a>
#### 10.2.4.2. `bool`
`boolean`の略で、日本語では論理値型という。`True`か`False`の2値しか取らない型である。<br>
各型の`__bool__`については [インスタンスが`bool`(真偽値)を持つ ― `__bool__()`メソッド](#one-zero-three-five)を参照。

<a id="one-zero-two-four-three"></a>
#### 10.2.4.3. `list`
順序付きの可変長配列。中の要素は任意の型を混在させることができる。<br>
`.append()`メソッドを用いて要素の数を増やすことができる。([属性(attribute)という概念](#one-zero-three-three))

```python
example = [1, 2, 3]
example.append(4)
print(example)  # [1, 2, 3, 4]
```
`[]`で囲んで作成する。代表的なイテラブルな型である。([配列とイテラブル](#one-zero-two-five))

<a id="one-zero-two-four-four"></a>
#### 10.2.4.4. `tuple`
順序付きの不変配列。`list`と異なり、作成後に要素を変更できない。

```python
example = (1, 2, 3)
example = ("a", "b", "c")
example = (1, "a", 3.14)
```
`()`で囲んで作成する。イテラブルな型である。([配列とイテラブル](#one-zero-two-five))

また、プログラミングとは直接関係はないが決まった個数の数値の組み合わせ（例えば座標`(x, y)`や`(a, b, c)`）をベクトルと呼ぶ。

<a id="one-zero-two-four-five"></a>
#### 10.2.4.5. `dict`
キーと値のペアを持つ連想配列。キーは文字列で、値は任意の型。
```python
example = {"a": 1, "b": 2, "c": 3}
example = {"name": "Taro", "age": 20}
```
`{}`で囲み、`key: value`の形で作成する。イテラブルな型である。[[#イテラブル(Iterable)な型]]

<a id="one-zero-two-four-six"></a>
#### 10.2.4.6. `set`
集合のような型で、重複を許さない(重複が除かれる)可変長配列である。<br>
下のコードのように`.add()`メソッドを用いて要素を加えることができる。([属性(attribute)という概念](#one-zero-three-three))
```python
example = {1, 2, 3}
example.add(4)
print(example)  # {1, 2, 3, 4}
```
`{}`では空の`set`ではなく空の`dict`を意味するため、空の`set`は`set()`関数で生成する。

<a id="one-zero-two-five"></a>
### 10.2.5. 配列とイテラブル
配列とは、複数の値をまとめて格納するデータ構造のこと。配列を持つ型は全てイテラブルな型であり、`for x in A` で値を1つずつ取り出せる。[[#イテラブル(Iterable)な型]]

プログラミングで複数の値をまとめて扱う方法は大きく3つに分類できる。

可変長配列 — `list`、`set`、`str`。作成後に要素の追加・削除・変更が可能。実務では最も頻繁に使われる。

不変配列 — `tuple`。作成後に変更できない。関数の複数の戻り値や、変更されてはいけないデータを扱うときに使う。

連想配列 — `dict`。キーと値のペアで、キーを使って値にアクセス。設定ファイルやデータベースの行のような、複数の関連したデータをまとめるときに使う。

<a id="one-zero-two-six"></a>
### 10.2.6. イテラブル(Iterable)な型
イテラブルな型と言うのはイテラブル型という単一の型ではなく、値を1つずつ取り出せる型の総称である。<br>
`list, tuple, dict, set, str`などが挙げられる。また、`for x in A`で`A`に入れることができる型は全てイテラブルな型である。<br>
そして、イテラブルな型のインスタンスの各要素を取り出して並べたときの並びをイテラブルという。

今後はイテラブルな型全てを意味するときには`Iterable[T]`と表記し、特定の型のイテラブルであることを強調するときには`Iterable[int]`などで表す。

また、
```
example = [1, 2, 3]
```
は`Iterable[int]`の中の`list[int]`型である。

<a id="one-zero-two-seven"></a>
### 10.2.7. `for`文の意味合い
`for`文は「繰り返し」と説明されることが多いが、厳密には**イテラブルな型の要素を1つずつ取り出して処理する** ことである。

```python
for x in example:
    print(x)
```

この場合、`in` の右側にはイテラブルな型（配列）が来る必要がある。<br>
`in` 演算子は「`example`の要素を1つずつ取り出す」という処理を行い、取り出された要素が `x` に代入されて、`for`ブロック内の処理が繰り返される。

つまり、`for x in A` が実行されるには、`A` がイテラブルな型である必要がある。`list, tuple, dict, set, str` などがイテラブルな型なので、これらに対して `for` 文を使うことができるのである。

<a id="one-zero-two-seven-one"></a>
#### 10.2.7.1. Iterableな型へのアクセス

``` python
example = (1, 2, 3) #tuple
example = {1, 2, 3} #set
example = {"a": 1, "b":2, "c": 3} #dict
example = "hello" #str
```
の5つの`example`に対して

``` python
for x in example: #exampleの中の要素を1つずつ取り出して表示する。
	print(x)
```
を実行した場合の出力を見てみる。
それぞれ以下のようになる。

<a id="one-zero-two-seven-four"></a>
#### 10.2.7.4. `enumerate(Iterable[T]) -> Iterable[tuple(index, value)`
イテラブルなインスタンスに対して、それぞれの値に対してindexを付けたタプル形式のイテラブルを返す。
```python
example = ["a", "b", "c"]
for index, value in enumerate(example):
    print(index, value)
```
上のコードの出力結果は以下の通り。

```
2 c
```

<a id="one-zero-two-eight"></a>
### 10.2.8. アノテーションと型チェック ― デフォルト値`None`の処理
アノテーション(処理には無関係だが変数や返り値の型を宣言しておくこと)は型チェックの対象となる。<br>
そして、型を厳密に扱う場合、 **`None`は`NoneType`型の(唯一の)値であり、デフォルト値として設定するときには注意が必要である。**

例えば、<br>
`def function(A:str = None) -> dict`<br>
(`str`型の`A`を受け取って`dict`を返す)
というのは厳密には間違っている。`None`は`NoneType`型であり、`str`型にはならないからであある。

厳密に正しく書くなら<br>
`def function(A:str | None = None) -> dict`が正しい。ここで、`|`は「または」を意味し、`str | None`のような表記をunionという。<br>
なお、Pythonの特殊な仕様として、`None`は型名は`NoneType`であるもののアノテーションでは型名を`None`と表記する。アノアテーションで型名を書かない唯一の例である。

<a id="one-zero-two-eight-one"></a>
#### 10.2.8.1. 宣言型の言語の場合のデフォルト値`None`の扱い
Pythonでは`str`型の変数にデフォルト値として`None`を指定できるが、`str`と`None`は異なる型なので静的型付け言語では矛盾する。例えば、Rustでは`str`か`None`の両方の値を取りうる場合、`Option<&str>`というラッパー型で表現する。

```Rust
fn greet(name: Option<&str>) { }
```

このように、複数の型を取りうることを明示的に型で表現する必要がある。

<a id="one-zero-three"></a>
## 10.3. オブジェクト指向と属性

<a id="one-zero-three-one"></a>
### 10.3.1. オブジェクト指向とは
オブジェクト指向(Object-Oriented Programming, OOP)とは、プログラムを構成する基本単位を「オブジェクト(データとそれに対する処理をまとめたもの)」として捉えるプログラミングパラダイムである。

Pythonはオブジェクト指向の言語である。

従来の手続き型プログラミングでは、データと処理が分離していた。例えば、`int`や`str`のインスタンスがあり、それらに対して関数を適用するという形式である。<br>
一方、オブジェクト指向では、「ある概念が本来持っているべき変数(データ属性)と処理(メソッド)を1つの単位にまとめる」ことで、現実世界の構造をコードに反映させる。プログラミング的には、**「あるクラス(型)がデータやそのクラスに適した処理を持つ」** という点で特徴的である。

例えば、銀行口座を考えると
- 手続き型：`balance`という値と`withdraw(balance, amount)`という関数を分けて管理
- オブジェクト指向：`BankAccount`というクラス(上位の概念)に残高というデータと、引き出し処理というメソッドを両方持たせる

このデータと処理をまとめたものを属性(attribute)と呼び、属性がオブジェクト指向の中心概念となる。データ属性（フィールド）とメソッド属性（関数）の2つからなる。

<a id="one-zero-three-two"></a>
### 10.3.2. オブジェクト指向の3大要素
オブジェクト指向には以下の3つの重要な要素がある。

1. カプセル化(Encapsulation)
データと処理を1つの単位にまとめ、外部からの直接アクセスを制限する。これにより、オブジェクト内部の状態を保護し、一貫性を保つ。

2. 継承(Inheritance)
既存のクラスを基にして、新しいクラスを作成する。共通部分を親クラスにまとめ、差分を子クラスに記述することで、コードの再利用と拡張を実現する。

3. ポリモーフィズム(Polymorphism)
同じメソッド名で異なる動作を実現すること。例えば、複数の型が同じ名前のメソッドを持つとき、呼び出し側はその型を気にせず同じメソッド名で処理できる。

<a id="one-zero-three-three"></a>
### 10.3.3. 属性(attribute)という概念
`A.B`はA(という名前のインスタンス)が持つBという属性を呼び出す。属性の種類が以下がある。<br>
特に重要なのはデータ属性と

| 属性 | 書き方, 例 | 意味 |
| --- | --- | --- |
| データ属性 | var.name | 値や変数(＝フィールド) |
| メソッド属性 | var.strip | 関数（そのオブジェクト専用の処理） |
| クラス属性 | MyClass.value | クラスレベルで定義される共通値。インスタンス属性により上書き可能 |
| モジュール内関数 | re.sub | `.py`をインスタンスとして、その中の関数をメソッドとして見なしたもの |

<a id="one-zero-three-three-one"></a>
#### 10.3.3.1. メソッド属性の具体例
メソッド属性は、見かけ上メソッド(+関数)の第1引数が省略されているように見える。<br>
つまり、`A.B(b, c)`は`B(A, b, c)`だと見なすと分かりやすい。<br>
よく使う`str`型のメソッドをいくつか挙げる。
```python
example = "  hello world  "
print(example.strip())      # "hello world"（前後の空白を削除）
print(example.upper())      # "  HELLO WORLD  "（大文字に変換）
print(example.replace("world", "python"))  # "  hello python  "（置換）
print(example.split())      # ['hello', 'world']（スペース区切りで分割）
```

これらのメソッドは全て`str`型のインスタンスに対してのみ呼び出せ、そのインスタンスに対して機能する。

メソッド関数はメソッドではなく単なる関数として適用することができる。<br>つまり、`A.B()`は`B(A)`に変換することができる。<br>
しかし、`B(A)`ができても`A.B()`ができない場合はある。例えば、<br>
`int("25")`(`int`型の25が返ってくる)は成立するが`"25".int()`は成立しない。これは`int()`は型変換をするものであり、`str`は`int()`という自身を型変換させるメソッドを持っていないからである。

<a id="one-zero-three-three-two"></a>
#### 10.3.3.2. メソッド属性とモジュール内関数の違い
Pythonでは`.py`ファイル(モジュールと言う)もインスタンスであり、そのときに`.py`内(モジュール内)の関数はそのモジュールのメソッド属性になる。
とはいえ、`.py`内で出てくる一般的なインスタンスに対してのメソッドではなく、`.py`に対してのメソッドであるため使い方は異なる。

```python
import re

text = "hello world"

# `.replace`はtextという名付けをされた"hello world"インスタンスに紐づいたメソッド
result = text.replace("world", "python")   # "hello python"

# re.sub()はreモジュールのメソッド属性であるが、textのメソッドではない。
result = re.sub(r"world", "python", text)  # "hello python"
```

<a id="one-zero-three-four"></a>
### 10.3.4. メソッドとモジュール内関数の違い：
- **メソッド属性** (`obj.method()`)：そのオブジェクト専用の処理。オブジェクトに紐付いている
- **モジュール内関数** (`module.function()`)：モジュール経由で呼び出す。汎用的な関数をまとめたもの

```python
class A:
    value = 1          # クラス属性（デフォルト参照先）

    def __init__(self):
        self.value = 1 # インスタンス属性（各インスタンスに設定）

# 動作例
a = A() #右辺で`A`クラスのインスタンスを作成し、`a`というラベルづけ
print(a.value)  # 1（インスタンス属性が優先）
print(A.value)  # 1（クラス属性の初期値）

A.value = 2     # クラス属性を変更
print(A.value)  # 2（クラス属性が変更）

b = A()         # 新しいインスタンスを生成
print(b.value)  # 1（__init__でインスタンス属性が1に設定されるため）
print(A.value)  # 2（クラス属性は2のまま）
```

<a id="one-zero-three-five"></a>
### 10.3.5. インスタンスが`bool`(真偽値)を持つ ― `__bool__()`メソッド
実はインスタンスはbool値を持っていると見なすこともできる。というのも、インスタンスは`__bool__()`メソッドを持っているため、任意のタイミングで`__bool__()`を呼び出して自身に関してのbool値を与えることができるからである。

各型の`__bool__()`メソッドは、「自分自身(インスタンス)が空でないか・ゼロでないか」を判定する処理を持っている。例えば
```python
a = 0
a.__bool__()    # False

a = 1
a.__bool__()    # True

a = ""
a.__bool__()    # False

a = "a"
a.__bool__()    # True

a = []
a.__bool__()    # False

a = [0]
a.__bool__()    # True  ← 中身が0でもリスト自体は空でないのでTrue

a = None
a.__bool__()    # False
```

| 型 | `False`になる場合 | `True`になる場合 |
|---|---|---|
| `int` / `float` | `0`, `0.0` | それ以外の数値 |
| `str` | `""` (空文字列) | それ以外の文字列 |
| `list` / `tuple` / `set` / `dict` | 空(`[]`, `()`, `set()`, `{}`) | 要素が1つ以上ある |
| `NoneType` | 常に`False` | (なし) |

<a id="one-zero-four"></a>
## 10.4. 演算子(operator)について
演算子とは、コードの中でそれを用いたときに特定の処理をする記号のことである。例えば、`a+b`における`+`は`a`に対して`b`を加えるという処理をさせる記号である。そして、 [[#属性(attribution)という概念]] に書いてある`a.b`というのは「`a`という(クラス)構造体の`b`という属性にアクセスする」という意味を持つ演算子である(属性参照演算子 という名称がある)。
また、`+`や`.`は処理が1つに決まっているが、演算子の中には「この記号はこの処理」というように1対1対応で処理が決まっているような演算子だけでなく、**前後の構文によってどのような処理をするかに決まる演算子もある*。**

なお、`=` は演算子ではない。というのも、`=`はあくまで名付け(参照するオブジェクトを指定)しているのみである。
また、辞書や集合を定義するときの`A= {}`の`{}`も演算子ではない。これはリテラル構文と呼ばれるものである。

演算子の例としては以下がある。

**`[]`については僕の理解が浅いので間違ったことを書いている可能性があります。**

| 演算子  | 処理 |
| --- | --- |
| `.`  | 構造体の属性にアクセスする | 
| `[]` | リストを定義したり辞書構造体やリストの要素を取り出したり、新たに追加する。<br>[i]`A["happy"]`だと`A`という辞書構造体の`happy`というキーのアイテムを取り出す。<br>[ii]`A["happy"]= 2`だと`A`という辞書構造体に`happy`というキーとそれに対応する値`2`を追加する。<br>[iii]`A[n]`だとリストの`n+1`番目の要素にアクセスする。<br>[iv] `A= [a, b, c]`だと`A`をリストオブジェクトだと定義する。 |
| `==` | 後ろのオブジェクトを前のオブジェクトと比較して一致していた場合に`true`を、一致していない場合に`false`を返す。 |

<a id="one-zero-four-one"></a>
### 10.4.1. 演算子と関数
演算子を「特定の処理をする記号」と説明したが、「特定の処理をする」と聞いて思いつくのは関数ではないだろうか。<br>
実際、演算子というものは内部的にはメソッド関数の呼び出しをしているのである。<br>

例えば<br>
`c = a + b`というのは<br>
「`a`という名前が指すインスタンスが数字に関する型であり、そのインスタンスが`__add__()`メソッドを持っているとき、`+`の後ろにあるオブジェクトを引数として_`a.__add__(b)`を行う」という意味なのである。

なお、属性参照演算子の`.`は`__getattribute__()`メソッドであるが、パース(演算子をメソッドに直す作業が含まれる)をしてから再度呼び出しをすることはなく、<br>
`obj.A = obj.__getattribute__("A")= obj.__get_attrbute__(__getattribute__(A""))`のように再帰的に`.`が展開されることはない。

そして、演算子が関数なのだとしたら、関数は返り値を持つのだから演算子も返り値を持つはずである。

例えば、比較演算子`==`は「同じかどうかを確かめる」というのは国語的には正しいが、プログラミングの内部処理を表現していない。<br>

プログラミング的には「`==`の左右が同じインスタンスを指しているかを比べて、同じなら`True`を、異なるなら`Flase`を返す(つまり返り値の型は`bool`)」という役割をしている。

その他の`>`や`<`も`!=`も全て「比べているもの」ではなくて「比べた結果`bool`値、つまり`True`か`False`を返すもの」なのである。

なお、 `not`も演算子であり、機能は後ろにある様々なオブジェクトの持つ`x.__bool__()`を呼び出して、そこで返ってくる`True, False`を反転させるというものである。<br>
実は`not`は対応するメソッド名があるわけではない。インタープリタが直接利用している。

## 12.2. Chromiumってよく聞くのはなんで？
で、なぜChromiumという用語をよく聞くのかの話になりますが、それは先ほどしたChromiumの説明「ブラウザがブラウザとして動作するのに必要な様々な機能が入ったソフトウェア基盤」というのから来ています。

ブラウザはソフトウェアです。Chromiumはその基盤なので、ブラウザの基盤ということです。つまり、「ブラウザを作りたいなー」ってなったときにChromiumを使うようにすれば、主要な機能はすぐに実装できるわけです。

それこそGoogle Chromeは右上の方に「・」が9個並んだようなボタンがあり、そこを押すとGoogle系のサービスが色々出てくるじゃないですか。<br>

つまり、Chromiumを使えば基盤の機能は簡単に実装でき、逆に色々な拡張機能がChromiumを前提として作っているのもあり、**多くのブラウザがChromiumを基盤としています。**

### 12.2.1. Chromium系と非Chromium系
これまでの話から分かると思いますが、Google Chrome以外にもChromiumを基盤としたブラウザが非常に多くありまず。そのため、ブラウザを分類するときには「Chromium系」と「非Chromium」系の分岐がまず行われるわけです。
Chromium系と非Chromium系の有名なブラウザをいくつか紹介します。

- Chromium系: Chrome, Edge, Vivaldi, Brave, Opera などその他多数
- 非Chromium系: Firefox, Safari
  - Firefox（レンダリングはGecko、JSはSpiderMonkey）
  - Safari（レンダリングはWebKit、JSはJavaScriptCore）

### 12.2.2. ブラウザにメモリや速度の差が出ない理由

プログラミング言語には100倍以上の速度の差が出ることもザラですが、ブラウザはそうでもないですよね。賢明な読者の人はもう分かったかもしれませんが、その理由について書いておきます。

- **みんなChromiumだから** : まあそりゃそうですよね。みんなが基盤に同じものを使ってたら大した差が出るわけないですよね。

- **web側に合わせて似たような設計になるから** : ブラウザがwebページの表示以外の役目があるとしても、主な役割がwebページの表示である以上、webページの仕様にあわせないといけません。で、webページがchromiumのような設計を期待して作られてるなら、FirefoxでもSafariでも設計をChrimiumっぽくしないといけない部分があるわけです。<br>設計の自由度がそこまで高くないということです。<br><br>
プログラミング言語はWeb系(JavaScript), ロボット(C++, C#), AI・機械学習(Python), 科学計算(R)のように「この言語はこれに用いる」みたいなのを決めれて、向かないタスクは他の言語でカバーすれば良いので 設計に幅を出しやすく、その結果速度に100倍とかの差が出るんです。<br>
そもそもプログラミング言語は「開発しやすさ」と「動作の安定性」みたいな軸も入ってきますが、ブラウザに「開発のしやすさ」なんて軸はないですからね。
