# 15. その他の色々な話

<a id="15-1"></a>
## 15.1. APIとSDK

<a id="15-1-1"></a>
### 15.1.1. API, SDKの定義

現代のソフトウェア開発では、すべての機能をゼロから実装することはほぼなく、他のサービスやOSが持つ機能を再利用するのが一般的である。
そのような再利用を実現するための仕組みがAPIであり、APIを手軽に扱えるようにツールやライブラリをまとめたものがSDKである。

**大規模なプロジェクトになってくるほど「このプロジェクトで使うべきAPIやSDKはあるか？」の技術調査が重要になってくる。**

- API（Application Programming Interface）: あるソフトウェアが持つ機能を、外部のプログラムから呼び出すための窓口。「何をどう呼び出せば何が返ってくるか」を取り決めたインターフェース仕様のこと。

- SDK（Software Development Kit）: 特定のプラットフォームやサービス向けのアプリを開発するために必要なツール・ライブラリ・ヘッダファイル・ドキュメント等をひとまとめにしたパッケージ。

<a id="15-1-2"></a>
### 15.1.2. APIの特徴

APIはSDKに比べて知名度が高く、プログラミングをしたことが無い人も知っている人が多いかもしれない。

また、APIは必ずしもインターネット接続を必要としない。
X APIやYouTube APIや最近増えているAIのAPIなどはWeb APIというHTTPを通じてインターネット上のサービスと通信する仕組みだが、「API」はあくまで内部で機能を実装してその窓口を提供するものでしかないため、OSに関わる処理をするためのAPIなどのローカルAPIも存在する。

<a id="15-1-2-1"></a>
#### 15.1.2.1. APIの具体例

- Gemini API（Google）: インターネット経由で利用するWeb API。AIの推論機能をHTTPリクエストで呼び出せる。
- Windows API: Windowsが提供するローカルAPI。ウィンドウの作成やファイル操作など、OS内部の機能をネットワーク通信なしに直接呼び出せる。

<a id="15-1-3"></a>
### 15.1.3. SDKの特徴

SDKはAPIを内包していることが多い。たとえばWindows SDKにはWindows APIの定義（ヘッダファイル等）が含まれており、SDKを導入することでそのAPIを利用できるようになる。
APIが「呼び出し口の仕様」であるのに対して、SDKは「開発に必要なもの一式」という違いがある。

<a id="15-1-3-1"></a>
#### 15.1.3.1. SDKの具体例

- Claude SDK（Anthropic）: Claude APIをより簡単に扱うためのSDK。APIキーの管理やリクエストの送受信、ストリーミング処理といった定型作業をまとめて提供する。
- Windows SDK: Windows向けアプリ開発に必要なツール・ヘッダファイル・ライブラリをまとめたSDK。Windows APIの利用に必要な定義がここに含まれる。

## 15.2. ブラウザがブラウザとして動くために ― Chromium
ブラウザの話をすると絶対にChromiumの話が出るのでその説明をします。
まず、Chromiumは、Google主導で開発されている「ブラウザがブラウザとして動作するのに必要な様々な機能が入ったソフトウェア基盤」を開発するプロジェクトであり、かつそのプロジェクトで開発されている成果物でもあります。<br>
また、Chromiumはオープンソースで開発されています。<br>
今後は成果物としてのChromiumの話をします。

ここで、「ブラウザがブラウザとして動作するのに必要な機能」について話しておきます。
- 取得（ネットワーク）: URL解決、HTTP/HTTPS通信、キャッシュ、Cookie、証明書検証。まあネットワークの云々をやってると思ってください。

- 解釈と表示（レンダリング）: HTML/CSS解析、レイアウト計算、描画。あくまでwebページは`.html`や`.css`に「こういう感じで表示して」と書いているだけで、それを読み込んで実際に画面を出すのはブラウザの仕事です。

- 動的処理（JavaScript実行）: イベント処理、DOM操作、非同期処理。「このボタンを開いたらメニューを出して」などは`.js`や`.ts`に記述されますが、それを実際に読み込んでその通りに動作させ画面に反映させるのはブラウザの仕事ですよね。

- 画面と操作（ブラウザUI）: タブ、アドレスバー、履歴、ブックマーク、ダウンロード管理。それぞれのwebページの動作ではなく、ブラウザ全体としてのUIの話です。

- セキュリティ: サンドボックス、同一オリジン制約、権限制御。マイクとかカメラにアクセスしようとするサイトがあったときにダイアログを出すのもブラウザがやってますよね。

- 安定性（プロセス管理）: タブ分離、クラッシュ影響の局所化、GPUプロセスとの分担

- 拡張性（Web機能/API）: Storage、Service Worker、WebRTC、拡張機能API。Google Chromeは多くの拡張機能がありますが、それは「これをしたらGoogle Chrome上でこう動作する」というAPIがあるからですよね。

まあ後半の方はいまいちピンときません(僕もピンときません)が、まあとりあえずブラウザというのが色々とやっているというのは「確かに」ってなったと思います。

そして、この様々な機能を実際に行っているのが下の「コンポネント」と言われるやつになります。`Blink`と`V8`だけ覚えておけばいいっぽいです。
- `Blink`（レンダリングエンジン）: HTML/CSSの解釈、レイアウト、描画
- `V8`（JavaScriptエンジン）: JavaScript実行、JIT、GCなど
- マルチプロセス構成:
  - Browser process: タブ管理、権限、UI、全体統括
  - Renderer process: 各タブ/サイトの表示とJS実行（主にBlink/V8）
  - ほか: GPU process、Network serviceなど（バージョンにより構成は変化）
- サンドボックス: Renderer側を強く隔離して、侵害されても被害を限定する設計・

<a id="15-2-1"></a>
### 15.2.1. Chromiumってよく聞くのはなんで？
で、なぜChromiumという用語をよく聞くのかの話になりますが、それは先ほどしたChromiumの説明「ブラウザがブラウザとして動作するのに必要な様々な機能が入ったソフトウェア基盤」というのから来ています。

ブラウザはソフトウェアです。Chromiumはその基盤なので、ブラウザの基盤ということです。つまり、「ブラウザを作りたいなー」ってなったときにChromiumを使うようにすれば、主要な機能はすぐに実装できるわけです。
あとはそのブラウザ独自の設定を載せていけば、ブラウザが完成するわけです。

それこそGoogle Chromeは右上の方に「・」が9個並んだようなボタンがあり、そこを押すとGoogle系のサービスが色々出てくるじゃないですか。<br>
あれはChromiumの外部でGoogle Chromeが独自で載せた設定ですよね。

つまり、Chromiumを使えば基盤の機能は簡単に実装でき、逆に色々な拡張機能がChromiumを前提として作っているのもあり、**多くのブラウザがChromiumを基盤としています。**

<a id="15-2-1-1"></a>
#### 15.2.1.1. Chromium系と非Chromium系
これまでの話から分かると思いますが、Google Chrome以外にもChromiumを基盤としたブラウザが非常に多くありまず。そのため、ブラウザを分類するときには「Chromium系」と「非Chromium」系の分岐がまず行われるわけです。
Chromium系と非Chromium系の有名なブラウザをいくつか紹介します。

- Chromium系: Chrome, Edge, Vivaldi, Brave, Opera などその他多数
- 非Chromium系: Firefox, Safari
  - Firefox（レンダリングはGecko、JSはSpiderMonkey）
  - Safari（レンダリングはWebKit、JSはJavaScriptCore）

<a id="15-2-1-2"></a>
#### 15.2.1.2. ブラウザにメモリや速度の差が出ない理由

プログラミング言語には100倍以上の速度の差が出ることもザラですが、ブラウザはそうでもないですよね。賢明な読者の人はもう分かったかもしれませんが、その理由について書いておきます。

- **みんなChromiumだから** : まあそりゃそうですよね。みんなが基盤に同じものを使ってたら大した差が出るわけないですよね。

- **web側に合わせて似たような設計になるから** : ブラウザがwebページの表示以外の役目があるとしても、主な役割がwebページの表示である以上、webページの仕様にあわせないといけません。で、webページがchromiumのような設計を期待して作られてるなら、FirefoxでもSafariでも設計をChrimiumっぽくしないといけない部分があるわけです。<br>設計の自由度がそこまで高くないということです。<br><br>
プログラミング言語はWeb系(JavaScript), ロボット(C++, C#), AI・機械学習(Python), 科学計算(R)のように「この言語はこれに用いる」みたいなのを決めれて、向かないタスクは他の言語でカバーすれば良いので 設計に幅を出しやすく、その結果速度に100倍とかの差が出るんです。<br>
そもそもプログラミング言語は「開発しやすさ」と「動作の安定性」みたいな軸も入ってきますが、ブラウザに「開発のしやすさ」なんて軸はないですからね。
