# 11. 実際にプログラミングをするときの設計
プログラミングを安全にするための設計の話をしていきます。<br>
これはAIの活用に限らないもっと基礎的な話になります。<br>

<a id="11-1"></a>
## 11.1. プロジェクトごとにdirを分ける ― divide and conquer
最も基本的な話です。あの「ゆるコンピューター科学ラジオ」で前はめっちゃよく出てきていた「divide and conquer」の実践です。

例えば、home dirが下みたいになってるなら、ここに`dev/`みたいなのを作って、「`~/dev/`の下でプログラミングをする」みたいに決めてしまえばいいわけです。

```text
/C:/Users/<ユーザー名>/ (= ~)
├─ Desktop
├─ Documents
├─ Downloads
├─ Music
├─ Pictures
├─ Videos
├─ AppData
│  ├─ Local
│  ├─ LocalLow
│  └─ Roaming
├─ OneDrive
├─ .local (最初の状態では存在しない)
│  └─ bin  
└─ (そのほかユーザーが作成したフォルダ)
```

そして、その中でGitHubのアカウントごとに分けた上で、その下でプロジェクトごとに分けましょう。

本当の推奨で言えばghqというのを導入することです。このghqは上の「アカウントごとに分けてからプロジェクトごとに分ける」を自然に行えます。

`~/dev/`をルートとした場合、ghqを使うと次のような構造になります。

```text
~/dev/
├─ github.com/
│  ├─ hirose-suguru/
│  │  ├─ pdf-summarizer/
│  │  ├─ timetable-scraper/
│  │  ├─ dotfiles/
│  │  └─ til/
│  └─ alan-smithee/
│     ├─ recipe-manager/
│     └─ workout-log/
```

このように、`github.com` → アカウント名 → プロジェクト名、という3階層の構造でリポジトリが整理されます。

<a id="11-2"></a>
## 11.2. プロジェクトの中でソースコードを置く場所を決める

人間が直接呼び出すスクリプト(ファイル)は、あまり深い場所に置きたくありません。例えば呼び出すファイルが`very/deep/deeper/deepest/test.py`だと打つのも大変です。かといってロジックもプロジェクトディレクトリの直下に散らかしたくはないです。

ということで、呼び出し部分は「呼び出しの責務だけを担うもの」として、プロジェクトディレクトリの直下に置いて、実際のロジックは`src/`や`modules/`などのサブディレクトリに分けて置きます。これは「呼び出す」という責務と「計算・処理をする」という責務を分離する設計で、責務の分離の一例と言えます。これは後で説明します。

たとえばghqのツリーに出てきた`pdf-summarizer/`というプロジェクトであれば、次のような構造になります。`src/`の中にロジックを書くと言いましたが、その中でも「これはこういう処理をする」というのでさらにdirを分けておくと良いです。

```text
pdf-summarizer/
├─ run.py          #人間やAIは(uv run)python run.py を実行する。
└─ src/            # ロジックはこっちに書く。
   └─ タスク名/    # タスクごとにさらにdirを分けるとより良い。
       └─ ... 
```

<a id="11-3"></a>
## 11.3. 設定ファイルを決める ― tomlがおすすめ

「このロジックは絶対に変わらない」というもの以外は、基本的に設定ファイルで管理することを推奨します。ロジックを書くファイルの中にファイルパスや固定値を直接書く(これをハードコードと言います)のは良くないです。

たとえば
- 「入力・出力のファイル名は何か」
- 「ある機能をオンにするかオフにするか」

といったものは、設定ファイルで管理するべきです。<br>
特に出力ファイルはAIがテスト目的で勝手に新しいファイルを作ろうとすることがあります。設定ファイルで出力先を管理して、それもAIに徹底させると、そういった意図しないファイル生成を減らすことにもつながります。

設定ファイルの形式にはtoml、json、yaml、ini、jsonl、json5などがありますが、人間が直接触るような設定ファイルにはtomlを使うのがよいです。圧倒的に書きやすく読みやすい形式で、Pythonであれば`tomlkit`というライブラリで簡単に扱えます。<br>
jsonは有名ですしその高機能版であるjson5もありますが、人間が読み書きしづらいですし括弧({})の影響で文字数が多くなりがちです。人間が全く触らないであろうキャッシュのファイルなどに使うのが良いと思います。

```text
pdf-summarizer/
├─ run.py
├─ src/
│  └─ ...
└─ config.toml     ← 入出力ファイル名や機能フラグをここで管理
```

また、場合によって設定が一気に変えてテストしてみたくなりそうなタスクなら「どの設定ファイルを読み込むか」の設定ファイルを持っておくと良いと思います。例えば`active_toml.toml`に「どの設定ファイルを読み込むか」の設定を書く感じです。<br>
つまり、処理としては「まずはactive_toml.tomlを読んで、そこに書いてあるファイルの設定を読み込む」という2段階の読み込みにするということです。
```text
pdf-summarizer/
├─ run.py
├─ src/
│  └─ ...
├─ config/
│  ├─ jan_config.toml # 1月用の設定
│  └─ feb_config.toml # 2月用の設定
└─ active_toml.toml   # こっちでjan_config.toml や feb_config.toml とそのときの都合に合わせて書き換える。
```

### 11.3.1. tomlの設定方法 基礎
tomlの最も基本的な設定方法を説明します。例えば、`test_config.toml`が
``` toml
[paths]
input_file = "input_test.txt"
output_file = "output_test.txt"

[features]
cache = True # キャッシュ
max_retries = 3
```

上のようになっているとします。`[paths]`や`[features]`が設定項目のグループで、`input_fle`や`cache`が個別の設定項目です。

このように、グループ分けをしたうえでさらに設定が分かれている状態をネストと言います。(ネストは本来はもっと広い概念で、色んなところで使われますがとりあえずこれでいきます)

そして、この`test_config.toml`のパスを`toml_path`として与えておけば
``` python
import tomlkit

with open(toml_path, encoding="utf-8") as f:
          config_data = tomlkit.parse(f.read())

data = tomlkit.parse(f.read())
        return Config(
            input_file=Path(config_data["paths"]["input_file"]),
            output_file=Path(config_data["paths"]["output_file"]),
            enable_cache=config_data["features"]["cache"],
            max_retries=config_data["features"]["max_retries"],
```
のみたいな感じに書けます(これはイメージとしてのコードなので、これでは動きません)。

`config_data["paths"]["input_file"]`の最初の`["paths"]`がtomlのグループ, 2つ目の`["input_file"]`がグループの中の個別の設定ということです。
今は1段階のネストしかしていませんが、`.`を使えばもっと深くネストすることもできるので、自分のやりたいようにやってみましょう。

<a id="11-4"></a>
## 11.4. 責務の分離 ― 1責務で1ファイル
本来は1つの責務に対して1つのファイルが理想です。何を「1つの責務」と見なすかは難しいですが、まずは次の区分を意識するとよいです。<br>
「読み込み」「文字列などの整形」「計算」「ファイルへの出力」「動作の流れをまとめたオーケストレーター（呼び出しスクリプトから呼ばれる）」「呼び出し」<br>
これらはそれぞれ責務が異なるので、ファイルとして分離させるのが望ましいです。

```text
pdf-summarizer/
├─ run.py                   ← 呼び出し（エントリーポイント）
├─ src/
│  ├─ main_flow.py          ← 動作の流れをまとめたオーケストレーター
│  ├─ fetch/                ← PDFの取得に関わるタスク
│  │  ├─ loader.py          ← 読み込み
│  │  └─ formatter.py       ← 整形
│  └─ summarize/            ← 要約に関わるタスク
│     ├─ summarizer.py      ← 計算・処理
│     └─ writer.py          ← 出力
└─ config.toml
```

`src/`の中でもタスクが増えてきたらさらにディレクトリを切って整理するとよいです。ファイルが増えても`src/`直下が散らからずに済みます。

<a id="11-4-1"></a>
### 11.4.1. 副作用 ― 値を返す純粋関数, 外部の状態を変える副作用関数

関数というと数学での関数とプログラミングの関数が全然違うように感じるかもしれませんが、数学的な意味での関数（同じ入力に対して必ず同じ値を返す関数）は、プログラミングの世界では「純粋関数」と呼ばれます。

一方で、ファイルの読み書き（ファイルI/O）や実行中プロセスの状態変更など、外部の状態を変える処理のことを「副作用」といいます。このドキュメントではそういった処理を行う関数を「副作用関数」と呼びます。

責務の分離によるファイル分けは今後の設計を上手く予想しないと難しい部分もありますが、それならば1つのファイル内で関数を上手く分けることから始めましょう。<br>
まずは「ファイルの読み込みをする関数」「計算をする関数」「出力やファイルの読み書きをする関数」の3種類に分けるレベルから始めるとよいです。もちろん、それぞれをファイル単位で分けてもかまいません。

<a id="11-5"></a>
## 11.5. クラス構造を起点にする
特にAIとの相性が良い部分だと思います。簡単に言うと「同じグループの変数と、その変数を使うような汎用的な処理をまとめておく」ということです。
基本的に機能は後から拡張されていくものなので、まず「このタスク専用のクラス」と「このタスクにも使えるが他の用途でも使えそうな汎用的なクラス」を先に設計することから始めると良いです。よく使うデータをフィールド（データ属性）として持ち、よく使う処理をメソッドとして定義しておきます。

汎用的なクラスは特定のタスクのファイルに一緒に書いてしまうと、そのファイルが読まれなかった場合にクラスの存在に気づかれず、そのクラスを前提とした設計が組まれにくくなります。そのため、汎用クラスをまとめて置く専用のディレクトリを用意しておくとよいです。

```text
pdf-summarizer/
├─ run.py
├─ src/
│  ├─ models.py     ← 汎用クラスをまとめる
│  └─ ...
└─ config.toml
```

AIに対しては「辞書でのデータの引き回しが無いように」みたいなことを言えば、良い感じにクラスを活用した設計をやってもらいやすくなります。

また、特にtomlに書く設定みたいな部分はクラス化しておくと良いかもしれませんね。例えばこんな感じです。
```python
from dataclasses import dataclass
from pathlib import Path
import tomlkit

@dataclass
class TomlConfig:
    input_file: Path
    output_file: Path
    enable_cache: bool
    max_retries: int

    @staticmethod
    def from_toml(toml_path: str) -> "Config":
        with open(toml_path) as f:
            data = tomlkit.parse(f.read(toml_path, encoding="utf-8"))
        return Config(
            input_file=Path(data["paths"]["input_file"]),
            output_file=Path(data["paths"]["out_file"]),
            enable_cache=data["features"]["cache"],
            max_retries=data["features"]["max_retries"],
        )
```
こうすると`TomlConfig`クラスを見るだけで「今何が設定項目になっているのか」が分かりますし、また各フィールド(`input_file, enable_cache`)も型が定義されるので型安全という状態になります。(Pythonは動的型付けなので、もしそのフィールドに異なる型が入って実行されてもエラーにはならないですが、型チェックのときには弾かれるようになります。)

また、`from_toml`メソッドにより、このクラスの中で「設定を読み込む」という処理が完結します。今後設定項目が増えても、このクラスのフィールドとメソッドに追加すれば良いだけです。これもクラスを設計した利点です。

ちなみに、僕はAIに一発で「これが何か」というのが伝わるのが良いと思っているので`modules/`(上の図の`src/`)の中に`classes/`を作って、その中にさらにタスクごとにdirを作ってます。また、汎用的なクラスはそれだけで300行とかになったりするので、汎用的なクラス1つに対して1つのファイルを用意したりもします。
